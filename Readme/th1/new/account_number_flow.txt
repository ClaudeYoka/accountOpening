Résumé rapide :

- Source principale du numéro de compte : champ du formulaire client (champ visible sur la page de saisie).
- Champ HTML(s) :
  - `Ecobank Account Opening Form Customer.html` — input principal : `<input type="text" id="bank-account-number">`
  - `cso/rib.php` — input pour générer un RIB imprimable : `<input type="text" id="account-number" name="account">`

Flux côté client :

1) Saisie / recherche
   - L'utilisateur peut taper un numéro dans `#bank-account-number`.
   - La page possède un bouton "Generate / Search" (implémenté dans `cso/ecobank_account_form.php`), qui lit la valeur :
     `var q = document.getElementById('bank-account-number').value.trim();`
   - Ensuite le client envoie `q` en POST à `search_compte.php` :
     `fetch('search_compte.php', { method: 'POST', body: 'q='+encodeURIComponent(q) })`
   - `search_compte.php` va chercher une entrée dans la table `tblCompte` par plusieurs colonnes (id, emp_id, nip, id_num, mobile1, mobile2, email). Si trouvé, il renvoie la ligne du compte (JSON) pour pré-remplir la fiche.

2) Soumission du formulaire
   - Quand le formulaire est soumis (par POST JSON ou form-encoded), le serveur reçoit les données dans `cso/save_ecobank_form.php`.

Flux côté serveur (enregistrement) :

- Tables / base :
  - Connexion via `includes/config.php` (constante `DB_NAME` si définie).
  - Les données de la soumission sont enregistrées dans la table `ecobank_form_submissions`.
  - Si la table n'existe pas, `save_ecobank_form.php` la crée (voir CREATE TABLE initial ci-dessous).

- Comment le numéro est récupéré dans le payload :
  - La fonction utilitaire `pick_first_nonempty($data, [...])` teste plusieurs clés dans cet ordre et retourne la première valeur non vide :

```php
$account_number = pick_first_nonempty($data, ['bank-account-number', 'bank_account_number', 'account_number', 'account-number', 'customer_id']);
$bank_account_number = pick_first_nonempty($data, ['bank-account-number','bank_account_number']);
```

  - Cela permet d'accepter les différentes variantes de noms de champs envoyées par le formulaire ou par des serializations (ex: `bank-account-number`, `bank_account_number`, `account_number`, etc.).

- Insertion en base :
  - Le script construit un tableau `$toInsert` contenant entre autres `account_number`, `bank_account_number` et `data` (JSON normalisé).
  - Avant l'INSERT, il vérifie les colonnes existantes dans `information_schema` et n'inclut que celles qui existent pour éviter les erreurs sur des anciennes bases.
  - Si aucune colonne indexable n'est disponible, fallback : INSERT uniquement `(customer_id, data)`.

Extrait CREATE TABLE (simplifié) présent dans le fichier :
```sql
CREATE TABLE IF NOT EXISTS ecobank_form_submissions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    customer_id VARCHAR(128) DEFAULT NULL,
    account_number VARCHAR(128) DEFAULT NULL,
    bank_account_number VARCHAR(128) DEFAULT NULL,
    data JSON NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

Remarques sur la génération du numéro
------------------------------------
- Le système génère désormais automatiquement un **numéro de compte** lors de l'enregistrement client via `account_part` (implémenté dans `trackingController.php`).
  - Format : préfixe `3715` + suffixe numérique zéro-pad pour atteindre une **longueur totale de 11 chiffres** (3715 + 7 chiffres). Exemple : id 42 => `3715000042`.
  - Le numéro est stocké dans la colonne `tblCompte.account_number` (la colonne est ajoutée automatiquement si manquante).
  - **La recherche supporte maintenant `account_number`** : `search_compte.php` peut chercher par `account_number`, `id`, `id_num` ou `nip`.
- `rib_lookup.php` utilise maintenant d'abord `ecobank_form_submissions.account_number`/`bank_account_number`, puis, en fallback, recherche un `tblCompte` avec ce `account_number` et tente de retrouver une soumission correspondante (par account_number, bank_account_number, customer_name, mobile ou email).
- Pour le RIB, `rib_lookup.php` compose l'IBAN en concaténant des codes fixes (`country_code`, `bank_code`, `branch_code`) avec `account_number` et un éventuel `rib_key` fourni :
```php
$account_obj['iban'] = $account_obj['country_code'] . ' - ' . $account_obj['bank_code'] . ' - ' . $account_obj['branch_code'] . ' - ' . $account_obj['account_number'] . ($iban_key ? ' - ' . $iban_key : '');
```

Fallback / compatibilité / historique
-------------------------------------
- Les colonnes `account_number` et `bank_account_number` peuvent être ajoutées automatiquement par le script (ALTER TABLE) si manquantes — les logs `save_ecobank_form_debug.log` montrent ces opérations.
- Si une colonne n'existe pas à l'écriture, la valeur reste toujours disponible dans le JSON `data` (champ `data`) et pourra être backfillée plus tard si besoin.

Actions possibles (recommandations)
-----------------------------------
- Si tu veux que le système **génère** automatiquement des numéros de compte :
  - ajouter une routine serveur qui produit des numéros uniques (ex : séquence + préfixe),
  - stocker le numéro dans `ecobank_form_submissions.account_number` et (si souhaité) dans `tblCompte` pour permettre la recherche via `search_compte.php`.
- Si tu veux forcer la migration/backfill : écrire un script qui lit la valeur dans `data` (JSON) et met à jour `account_number` pour les lignes où elle est manquante.
